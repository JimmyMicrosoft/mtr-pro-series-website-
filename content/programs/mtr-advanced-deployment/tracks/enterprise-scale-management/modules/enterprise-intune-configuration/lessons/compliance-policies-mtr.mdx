## Overview

Compliance policies in Intune define what a "compliant" device looks like — and non-compliant devices can be blocked from accessing resources via Conditional Access. For MTR devices, compliance policies must be designed carefully: they should enforce meaningful security requirements (BitLocker enabled, Defender active, OS version current) without triggering requirements that break room functionality (like MFA). This lesson covers designing MTR compliance policies and integrating them with Conditional Access correctly.

---

## MTR Compliance Policy Design

**What to include in an MTR compliance policy:**

| Setting | Recommended Value | Reason |
|---------|------------------|--------|
| Minimum OS version | Current - 2 (e.g., 22H2 if current is 24H2) | Allows time for controlled OS update rollout; not too strict |
| BitLocker | Required | MTR devices should have BitLocker enabled (drives contain meeting data) |
| Secure Boot | Required | All current MTR hardware supports Secure Boot |
| Windows Defender — Real-time protection | Required | Defender should be active on MTR devices |
| Firewall | Required | Windows Defender Firewall should be active |
| Antivirus | Required | Defender or equivalent |
| Antispyware | Required | Defender |
| Code Integrity | Optional for rooms | TPM-backed code integrity; useful for high-security environments |

**What to exclude from an MTR compliance policy:**

| Setting | Reason to Exclude |
|---------|------------------|
| Password required / PIN | MTR devices use auto-sign-in via resource account; requiring a device PIN breaks this |
| Device Health Attestation — require healthy device | Transient compliance failures can block rooms; use grace period instead |
| Maximum minutes of inactivity before password | Room device; no user PIN to lock |

## Conditional Access and MTR Compliance

Conditional Access policies can require device compliance for resource access. For MTR devices:

**The problem:**
If a Conditional Access policy requires "device compliance" for access to Teams/Exchange resources, and an MTR device becomes non-compliant (e.g., Defender definition update check-in failure), the device may be blocked from signing in to Teams — taking the room offline.

**The solution — Conditional Access scoping for MTR:**
Create Conditional Access policies that apply to MTR room resource accounts with appropriate requirements:

```
Policy: MTR Device Access
Users/Groups: [MTR Resource Accounts group]
Cloud apps: Microsoft Teams, Exchange Online
Conditions:
  Device platforms: Windows
  Device state: Require compliant device: ON

Grant: Require device to be marked as compliant
Session: None
```

This policy requires device compliance for the MTR room accounts. The compliance policy must be attainable without MFA — which it is, because device compliance is evaluated at the device level, not the user level. MFA would require user interaction, which breaks MTR.

**Critical: Verify no MFA policy applies to room accounts:**
As established in Program 2, room resource accounts must be excluded from MFA policies. This remains true when adding device compliance requirements — requiring both MFA and device compliance would make the room accounts completely non-functional. The Conditional Access architecture for MTR is:

- MFA policy: Excludes MTR resource accounts group
- Device compliance policy: Applies to MTR resource accounts group (requires compliant device)

This enforces device compliance without requiring MFA.

## Compliance Grace Period

The compliance grace period allows devices to be non-compliant for a specified period before being marked non-compliant in Conditional Access. This prevents transient issues (a Defender definition update that temporarily shows outdated status, a momentary check-in failure) from blocking rooms.

**Configure grace period:**
In the compliance policy → Properties → Non-compliance actions:
- Action: Mark device non-compliant
- Schedule (days after non-compliance): **7 days**

With a 7-day grace period:
- A device that becomes non-compliant is not blocked by Conditional Access for 7 days
- The compliance report shows the device as "In Grace Period" — visible to IT for investigation
- If the device is still non-compliant after 7 days, it is marked non-compliant and Conditional Access blocks resource access

*For rooms in high-security environments, reduce the grace period. For standard commercial environments, 7 days is appropriate — it provides time for the IT team to identify and remediate genuine compliance issues before rooms go offline.*

## Compliance Status Interpretation

In Intune → Devices → Compliance → select the compliance policy → Device status:

| Status | Meaning | Action |
|--------|---------|--------|
| Compliant | Device meets all policy requirements | None |
| Not compliant | Device fails one or more requirements after grace period | Investigate and remediate |
| In grace period | Device fails one or more requirements, within grace period | Monitor; remediate before grace period expires |
| Not evaluated | Device has not checked in within the compliance check interval | Verify device is online in Pro Management; check Intune sync |
| Error | Compliance policy evaluation failed on device | Check Intune device sync status |

**Investigating non-compliant devices:**
- In Intune → Devices → select non-compliant device → Device Compliance → see which specific compliance setting is failing
- Common causes:
  - **OS version below minimum:** Device missed a Windows Update; check update policy assignment and rings
  - **Defender outdated:** Defender definitions not updating; verify network access to Windows Update endpoints
  - **BitLocker not enabled:** Device did not receive BitLocker configuration; check BitLocker Intune policy assignment

## Remediating Non-Compliant MTR Devices Remotely

Most compliance failures can be remediated without a site visit:

**Defender signature update:**
From Intune → Devices → select device → select "Sync" to force a policy sync and Defender update check. Or via PowerShell via Intune remediation:
```powershell
Update-MpSignature
```

**BitLocker not enabled:**
Deploy a BitLocker Intune configuration policy targeted to the MTR device group. BitLocker encrypts the OS drive with TPM-backed key protection — no user interaction required when configured for silent encryption via Intune.

**OS version below minimum:**
Review and update the Windows Update ring policy for the MTR device group (next lesson). Force a sync from Intune to push the update check.

## Summary

- **Compliance policy:** Include BitLocker, Defender, Secure Boot, OS version minimum. Exclude PIN, password, and device health attestation requirements that break room sign-in.
- **CA integration:** Require device compliance for MTR resource accounts; do NOT also require MFA (already excluded); these are independent requirements.
- **Grace period:** 7 days — prevents transient failures from blocking rooms; "In Grace Period" status is visible for monitoring.
- **Investigation:** Non-compliant devices show the specific failing setting in Intune; most common causes are Defender definitions, BitLocker not configured, or OS version lag.
