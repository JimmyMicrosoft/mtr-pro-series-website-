## Overview

An MTR device enrolled in Intune should receive the correct Windows configuration without conflicting with the organization's standard user-device policies. MTR devices are single-purpose appliances, not general-purpose Windows endpoints — they need a distinct policy set that configures them for dedicated room use. This lesson covers how to structure the policy hierarchy, what MTR-specific policies to apply, and how to avoid and resolve conflicts.

---

## Policy Architecture Principles

**Principle 1 — Separate MTR from user devices:**
Never apply the same Intune configuration profiles to both MTR devices and user workstations. MTR devices require different power settings, Windows Update behavior, network configuration, and security settings. Create a dedicated MTR device group and assign MTR-specific policies exclusively to that group.

**Principle 2 — Layer policies by scope:**
Structure policies in layers from most general to most specific:

```
Layer 1: Windows Security Baseline (applies to all Windows devices including MTR)
Layer 2: MTR Baseline (applies to all MTR devices — power, kiosk, update)
Layer 3: MTR Site-Specific (applies to MTR devices at specific sites — network, proxy)
Layer 4: MTR Room Type (applies to specific room types — boardroom vs. huddle)
```

**Principle 3 — Use settings catalog for available settings:**
Intune's Settings Catalog provides a structured interface for common Windows configuration. Use Settings Catalog profiles before resorting to OMA-URI for settings it covers — they are easier to audit and maintain.

**Principle 4 — Document OMA-URI sources:**
Custom OMA-URI paths are opaque — a path like `./Device/Vendor/MSFT/Policy/Config/Power/DisplayOffTimeoutPluggedIn` is meaningless without context. Document every OMA-URI policy with the source reference (Microsoft documentation link), the value set, and the reason.

## MTR Baseline Policy: Essential Settings

**Power management:**
MTR devices must not sleep or hibernate — this prevents the room from waking for scheduled meetings. Configure via Settings Catalog:

- `Power → Sleep → Allow Standby States When Sleeping On Battery`: **Disabled**
- `Power → Sleep → Allow Standby States When Sleeping On AC`: **Disabled**
- `Power → Hibernate → Allow Hibernate`: **Disabled**
- Display sleep is managed by the Teams Rooms app itself (not Windows power settings) — set Windows display timeout to "Never" to avoid conflict:

OMA-URI for display off timeout on AC:
```
OMA-URI: ./Device/Vendor/MSFT/Policy/Config/Power/DisplayOffTimeoutPluggedIn
Data type: Integer
Value: 0 (never)
```

**Remote Desktop:**
MTR devices do not need Remote Desktop for end users. However, IT administrators may need remote access for support. Configure:
- Enable Remote Desktop via Intune policy (Settings Catalog → Remote Desktop Services)
- Restrict RDP access to IT administrator accounts via firewall policy (Windows Defender Firewall → Inbound Rules → Remote Desktop)
- Disable Remote Desktop if the organization does not use it for MTR support

**Windows Defender Firewall:**
The MTR app requires specific firewall rules. Microsoft manages these through the Teams Rooms application itself, but enterprise Intune policies can conflict with app-managed rules. Configure:
- Do not apply overly restrictive enterprise firewall baseline policies to MTR devices that block Teams media ports (UDP 3478–3481)
- If applying a security baseline, verify the baseline does not block or restrict Teams media traffic

**Local administrator account:**
MTR devices have a local administrator account (`admin` by default on some OEM configurations). For enterprise management:
- Disable the built-in local administrator account via Settings Catalog or OMA-URI
- Create a named local admin account managed via LAPS (Local Administrator Password Solution) for break-glass access
- Ensure the IT support team has the LAPS password retrieval process documented

## MTR-Specific OMA-URI Policies

**QoS (DSCP marking) — applied in Program 2, recapped here in enterprise context:**

Enterprise MTR fleets use the same OMA-URI QoS policies from Program 2, but applied via Intune at scale:

```
Audio (50000–50019):
OMA-URI: ./Device/Vendor/MSFT/NetworkQoSPolicy/TeamsAudio/AppPathNameMatchCondition
Value: msteamsapp.exe

OMA-URI: ./Device/Vendor/MSFT/NetworkQoSPolicy/TeamsAudio/DSCPMarkingValue
Value: 46

OMA-URI: ./Device/Vendor/MSFT/NetworkQoSPolicy/TeamsAudio/SourcePortMatchCondition
Value: 50000-50019
```

At enterprise scale, verify QoS policy delivery via Intune reporting (see Lesson 5) — confirm all devices in the fleet show the QoS configuration as compliant.

**Proxy bypass (if applicable):**
If the organization uses a proxy and MTR devices need a bypass:

```
OMA-URI: ./Vendor/MSFT/NetworkProxy/ProxySettingsPerUser
Data type: Integer
Value: 0 (machine-wide proxy settings, not per-user)

OMA-URI: ./Vendor/MSFT/NetworkProxy/Exceptions
Data type: String
Value: *.teams.microsoft.com;*.lync.com;*.microsoft.com;*.skype.com
```

**Kiosk mode considerations:**
MTR Windows devices are NOT configured as Kiosk mode devices (Assigned Access) — the Teams Rooms app runs as a full Windows session managed by its own lock-down mechanism, not by Windows Assigned Access. Do not apply Assigned Access (Kiosk) policies to MTR devices; this will break the Teams Rooms application.

## Policy Conflict Resolution

When multiple Intune policies configure the same Windows setting, a conflict occurs. Intune reports conflicts in the device configuration profile status. Resolution:

**Identifying conflicts:**
In Intune → Devices → select device → Device Configuration → check for "Error" or "Conflict" status on any profile. Select the profile to see which specific setting is in conflict.

**Conflict resolution hierarchy:**
Intune does not have a strict "wins" hierarchy for conflicts between Configuration Profiles of the same type — both settings are marked as "Not configured (conflict)" and neither is applied. The resolution is to edit one of the conflicting profiles to remove the conflicting setting, or consolidate the settings into a single profile.

**Common MTR conflict scenarios:**

| Conflict | Cause | Resolution |
|----------|-------|------------|
| Power settings not applying | Enterprise Windows security baseline also sets sleep settings | Add MTR devices to an exclusion group for the enterprise power policy |
| Windows Update settings conflict | Enterprise update ring and MTR-specific update ring both target the device | Assign the MTR update ring only to the MTR device group; exclude MTR from the enterprise update ring |
| Firewall rules blocking Teams | Enterprise security baseline applies restrictive firewall rules | Add MTR device group exception to enterprise firewall policy, or create an explicit allow rule for UDP 3478–3481 |
| Proxy settings overriding bypass | Enterprise proxy policy and MTR bypass policy conflict | Consolidate into a single policy for MTR devices |

## Verifying Policy Application

After deploying configuration policies to MTR devices, verify application:

**Intune portal verification:**
- Devices → select device → Device Configuration
- Each assigned profile should show "Succeeded" status
- "Error" indicates a policy application failure; click for details and error code
- "Conflict" indicates a setting conflict with another profile

**On-device verification (for remote verification via PowerShell or Intune remediation scripts):**
```powershell
# Verify QoS policy is applied
Get-NetQosPolicy | Where-Object {$_.Name -like "Teams*"} | Select-Object Name, DSCPAction, AppPathNameMatchCondition

# Verify power settings
powercfg /query SCHEME_CURRENT SUB_SLEEP STANDBYIDLE
```

## Summary

- **Architecture:** Layer policies (baseline → MTR baseline → site-specific → room type); keep MTR device group separate from user workstation targeting.
- **Essential MTR settings:** Power (no sleep/hibernate), Remote Desktop (IT admin only), firewall (don't block Teams media).
- **OMA-URI:** Use for QoS DSCP marking, proxy bypass, settings not in Settings Catalog; document all OMA-URI with source and reason.
- **Kiosk mode:** Do NOT apply Windows Assigned Access to MTR devices — Teams Rooms manages its own lock-down.
- **Conflict resolution:** "Conflict" status means neither policy applies; fix by consolidating or scoping exclusions.
