## Overview

Azure Logic Apps and Power Automate share the same underlying runtime, but they serve different organizational contexts. Power Automate is optimized for M365-native workflows built and owned by IT professionals using a browser-based designer. Logic Apps is optimized for enterprise integration scenarios that require version-controlled deployment, custom connector development, complex branching logic, and higher execution volume. For an enterprise MTR fleet generating thousands of alerts per month, Logic Apps is often the more appropriate platform. This lesson covers the decision framework, a practical Logic App design for MTR incident routing, and the Service Bus pattern that makes webhook processing reliable at scale.

---

## Logic Apps vs Power Automate: Decision Framework

| Dimension | Use Power Automate | Use Logic Apps |
|-----------|-------------------|----------------|
| Primary users | IT admins, non-developers | Developers, architects |
| Deployment | Manual in browser | ARM templates, Bicep, CI/CD pipeline |
| Version control | Limited (export JSON) | Native via source control |
| Execution volume | < 500 runs/day | Thousands of runs/day |
| Custom connectors | Available, browser-built | ISE custom connectors, full API spec |
| Retry and error handling | Basic configure-run-after | Built-in retry policies, dead-letter queues |
| Pricing model | Per user or per flow/month | Consumption (per action) or Standard (fixed) |
| M365 connector access | Native (Teams, SharePoint, etc.) | Via managed connectors (same functionality) |
| Managed Identity | Not supported | Supported — no stored credentials |

For MTR automation that processes Pro Management webhooks at enterprise scale (fleet of 500+ rooms, potentially 100+ alerts/day), Logic Apps with Service Bus is the production-grade choice.

## Logic App Workflow Design for MTR Incident Routing

A Logic App workflow processes Pro Management webhooks and routes them to the appropriate handler:

```
[HTTP Trigger: POST /mtralert]
        │
        ▼
[Parse JSON] — extract incidentType, severity, deviceId, resourceUPN
        │
        ▼
[Switch — incidentType]
    ├── "Camera not detected"
    │   └── [Condition: severity == Critical]
    │         ├── YES → [HTTP: TAC restart endpoint]
    │         │         └── [Condition: restart HTTP 200?]
    │         │               ├── YES → [HTTP: add note to Pro Mgmt incident]
    │         │               └── NO  → [HTTP: create SNOW incident]
    │         └── NO  → [HTTP: create SNOW incident (NonUrgent)]
    ├── "Network connectivity"
    │   └── [HTTP: create SNOW incident → Network Ops group]
    ├── "Sign-in failed"
    │   └── [HTTP: create SNOW incident → Identity team]
    └── Default
        └── [HTTP: log to Azure Log Analytics — informational only]
```

In Logic Apps, this is implemented using the **Switch** action (like a `case` statement) with each case handling one incident type. Retry policies on HTTP actions automatically retry on transient failures (network timeouts, 429 throttling) with configurable count and interval.

## Configuring Retry Policies

Logic App HTTP actions support built-in retry policies without custom code:

```json
{
  "type": "Http",
  "inputs": {
    "method": "POST",
    "uri": "https://[instance].service-now.com/api/now/table/incident",
    "headers": { "Authorization": "@{concat('Bearer ', variables('SNOWToken'))}" },
    "body": "@body('Build_Incident_Payload')"
  },
  "runAfter": {},
  "runtimeConfiguration": {
    "staticResult": { "staticResultOptions": "Disabled" }
  },
  "operationOptions": "None",
  "retryPolicy": {
    "type": "exponential",
    "count": 4,
    "interval": "PT7S",
    "minimumInterval": "PT5S",
    "maximumInterval": "PT1H"
  }
}
```

The exponential retry policy retries up to 4 times with increasing backoff (5s, 7s, 14s, 28s...) before marking the action as failed. This handles transient ServiceNow API unavailability without manual intervention.

## Azure Service Bus for Reliable Webhook Processing

A direct webhook → Logic App connection has a single point of failure: if the Logic App is temporarily unavailable (during deployment, maintenance, or a platform outage), the Pro Management webhook is lost. Azure Service Bus acts as a durable buffer:

```
Pro Management
      │
      ▼ HTTP POST
Azure Function (lightweight receiver)
      │ enqueue message
      ▼
Azure Service Bus Queue
      │ dequeue (Logic App consumer)
      ▼
Logic App (processes message)
      │
      ▼
Remediation actions
```

The Azure Function receiver acknowledges Pro Management's webhook immediately (HTTP 200) and enqueues the payload to Service Bus. The Logic App reads from the queue at its own pace — if the Logic App is temporarily unavailable, messages wait in the queue (configurable retention: up to 7 days) and are processed when the Logic App comes back online.

Service Bus setup in Logic Apps:

```powershell
# Create Service Bus namespace and queue via PowerShell (Azure module)
New-AzServiceBusNamespace -ResourceGroupName "MTR-Automation-RG" `
    -Name "mtr-automation-sb" -Location "uksouth" -SkuName "Standard"

New-AzServiceBusQueue -ResourceGroupName "MTR-Automation-RG" `
    -Namespace "mtr-automation-sb" -Name "pro-mgmt-incidents" `
    -LockDuration "PT5M" `      # 5 minute message lock for processing
    -MaxDeliveryCount 5         # Dead-letter after 5 failed attempts
```

Messages that exceed `MaxDeliveryCount` (failed processing 5 times) are moved to the **dead-letter queue** — a separate queue that holds messages requiring manual investigation. Monitor the dead-letter queue count as an operational metric; a growing dead-letter queue indicates a processing failure pattern.

## Managed Identity for Logic Apps

Managed Identity eliminates stored credentials in Logic App definitions — the Logic App authenticates to Azure resources (Key Vault, Service Bus, Graph API) using its own identity:

```powershell
# Enable System-Assigned Managed Identity on Logic App
$logicApp = Get-AzLogicApp -ResourceGroupName "MTR-Automation-RG" -Name "mtr-incident-router"
Set-AzResource -ResourceId $logicApp.Id -Properties @{
    identity = @{ type = "SystemAssigned" }
} -Force

# Grant the Logic App's Managed Identity permission to read Key Vault secrets
$principalId = (Get-AzLogicApp -ResourceGroupName "MTR-Automation-RG" -Name "mtr-incident-router").Identity.PrincipalId
New-AzRoleAssignment -ObjectId $principalId `
    -RoleDefinitionName "Key Vault Secrets User" `
    -Scope "/subscriptions/$subscriptionId/resourceGroups/MTR-Automation-RG/providers/Microsoft.KeyVault/vaults/mtr-keyvault"
```

In the Logic App definition, reference Key Vault secrets via the Key Vault managed connector using the Managed Identity — no client secrets stored in the workflow definition or environment variables.

## Cost Optimization: Consumption vs Standard Plan

| Factor | Consumption Plan | Standard Plan |
|--------|-----------------|---------------|
| Pricing | Per action execution (~$0.000025/action) | Fixed monthly ($~150-300/month) |
| Break-even | ~6M actions/month | Above ~6M actions/month |
| MTR fleet of 200 rooms | ~50K actions/month → ~$1.25/month | Overkill |
| MTR fleet of 2000 rooms | ~500K actions/month → ~$12.50/month | Still Consumption |
| Managed Services (10K alerts/day) | ~3M actions/month → ~$75/month | Consider Standard |
| VNet integration | Not available | Available |
| Stateful + stateless workflows | Stateful only | Both |

For most enterprise MTR deployments (< 500 rooms, < 100 alerts/day), the Consumption plan is dramatically more cost-effective than Standard. The Standard plan becomes attractive at managed services scale with high alert volume and when VNet integration is required for private connectivity to on-premises ITSM systems.

## Summary

- **Decision framework:** Use Power Automate for M365-native, browser-managed workflows at moderate volume. Use Logic Apps when you need CI/CD deployment, version control, higher execution volume, or Managed Identity.
- **Incident routing:** Switch action on `incidentType` with nested conditions on severity provides clean, readable routing logic; exponential retry policies handle transient API failures automatically.
- **Service Bus buffer:** Azure Function → Service Bus Queue → Logic App consumer prevents webhook loss during Logic App deployments or outages; dead-letter queue surfaces messages that failed repeated processing attempts.
- **Managed Identity:** Eliminates stored credentials in Logic App definitions; grant the Managed Identity access to Key Vault, Service Bus, and other Azure resources via RBAC.
- **Cost:** Consumption plan is cost-effective for most enterprise MTR deployments; Standard plan makes sense only at managed services scale (> 6M actions/month) or when VNet integration is required.
