## Overview

The two Graph analytics APIs most relevant to MTR operations are the Reports API — which provides aggregate usage counts across device types — and the Call Records API — which provides detailed per-call quality metrics at the session and segment level. Together they enable programmatic quality reporting without requiring CQD portal access, allowing you to integrate quality data into your existing reporting pipeline. This lesson covers both APIs, with particular focus on the Call Records data model and how to aggregate it into actionable per-room quality metrics.

---

## Microsoft 365 Reports API for Teams

The Reports API provides pre-aggregated usage data across the M365 tenant. For MTR, the device usage report shows how many calls are being made from room devices:

```powershell
# Teams device usage distribution — how many users used each device type in the period
# Period options: D7, D30, D90, D180
$reportUri = "https://graph.microsoft.com/v1.0/reports/getTeamsDeviceUsageDistributionUserCounts(period='D30')"

# Reports API returns CSV by default
$csv = Invoke-RestMethod -Uri $reportUri -Method GET -Headers $headers

# Parse the CSV
$reportData = $csv | ConvertFrom-Csv
$reportData | Select-Object "Report Date", "Teams Rooms (Windows)", "Teams Rooms (Android)",
    "Teams Phone", "Web", "Windows", "Mac" | Format-Table
```

The `Teams Rooms (Windows)` and `Teams Rooms (Android)` columns show the count of MTR devices that participated in at least one meeting during the period. This is a tenant-wide aggregate — not per-room — but it confirms that the fleet is actively being used and gives a trend line when queried weekly.

For per-room meeting activity, the calendar-based utilization approach from the previous lesson is more granular than the Reports API.

## Call Records API: Data Model

The Call Records API returns structured call quality data at three levels of granularity:

```
Call Record
└── Sessions (each participant pair in the call)
    └── Segments (each media segment within a session)
        └── Media (audio, video, app sharing streams)
            └── Streams (inbound/outbound — contains quality metrics)
```

Each **stream** contains the network quality metrics: packet loss, jitter, and round-trip time. For MTR quality analysis, you are primarily interested in the audio stream metrics for the room device endpoint.

```powershell
# Query call records from the last 7 days
$since    = (Get-Date).AddDays(-7).ToUniversalTime().ToString("o")
$callsUri = "https://graph.microsoft.com/v1.0/communications/callRecords" +
            "?`$filter=startDateTime ge $since" +
            "&`$expand=sessions(`$expand=segments)" +
            "&`$top=100"

$callRecords = Invoke-GraphRequest -Uri $callsUri -Headers $headers
Write-Host "Retrieved $($callRecords.Count) call records"
```

<Callout type="note">
The Call Records API returns at most 100 records per page — use pagination (`@odata.nextLink`) for large result sets. Data is available approximately **15 minutes** after the call ends. The maximum lookback window in v1.0 is **60 days**.
</Callout>

## Extracting Quality Metrics from Call Records

Parsing the nested call record structure to extract audio quality metrics for each participant:

```powershell
function Get-CallQualityMetrics {
    param([array]$CallRecords)

    $metrics = [System.Collections.Generic.List[PSObject]]::new()

    foreach ($call in $CallRecords) {
        if (-not $call.sessions) { continue }

        foreach ($session in $call.sessions) {
            if (-not $session.segments) { continue }

            foreach ($segment in $session.segments) {
                if (-not $segment.media) { continue }

                foreach ($media in $segment.media) {
                    if ($media.label -ne "main-audio") { continue }

                    foreach ($stream in $media.streams) {
                        $net = $stream.network
                        if (-not $net) { continue }

                        # Identify if this endpoint is a Teams Room
                        $endpoint = if ($stream.streamDirection -eq "callerToCallee") {
                            $session.caller
                        } else {
                            $session.callee
                        }

                        $isRoom = $endpoint?.userAgent?.headerValue -like "*RoomDevice*" -or
                                  $endpoint?.userAgent?.headerValue -like "*TeamsRooms*"

                        $metrics.Add([PSCustomObject]@{
                            CallId          = $call.id
                            CallStart       = $call.startDateTime
                            StreamId        = $stream.streamId
                            IsRoomEndpoint  = $isRoom
                            PacketLossRate  = $net.packetLoss
                            AverageJitter   = $net.averageJitter
                            RoundTripTime   = $net.averageRoundTripTime
                            BandwidthLow    = $net.bandwidthLowEventRatio
                            JitterBad       = $net.jitterInterArrival -gt 30
                            PacketLossBad   = $net.packetLoss -gt 0.05
                        })
                    }
                }
            }
        }
    }

    return $metrics
}

$qualityMetrics = Get-CallQualityMetrics -CallRecords $callRecords
```

## Building Per-Room Quality Aggregates

Once you have individual stream metrics, aggregate them per room to identify chronic quality issues:

```powershell
# Filter to room endpoint streams only
$roomStreams = $qualityMetrics | Where-Object { $_.IsRoomEndpoint -eq $true }

# Aggregate by call origin (approximate per-room — requires correlating with room account)
$roomQuality = $roomStreams | Group-Object CallId | ForEach-Object {
    $streams = $_.Group
    [PSCustomObject]@{
        CallId           = $_.Name
        CallStart        = ($streams | Select-Object -First 1).CallStart
        AvgPacketLoss    = [Math]::Round(($streams | Measure-Object PacketLossRate -Average).Average, 4)
        AvgJitter        = [Math]::Round(($streams | Measure-Object AverageJitter -Average).Average, 1)
        AvgRTT           = [Math]::Round(($streams | Measure-Object RoundTripTime -Average).Average, 0)
        PoorCallFlag     = ($streams | Where-Object { $_.PacketLossBad -or $_.JitterBad }).Count -gt 0
    }
}

# Quality thresholds (aligned with CQD definitions)
$poorCalls = $roomQuality | Where-Object { $_.PoorCallFlag -eq $true }
$poorRate  = if ($roomQuality.Count -gt 0) {
    [Math]::Round(($poorCalls.Count / $roomQuality.Count) * 100, 1)
} else { 0 }

Write-Host "Poor call rate (room endpoints): $poorRate%"
Write-Host "Poor calls: $($poorCalls.Count) / $($roomQuality.Count)"
```

| Metric | Poor Quality Threshold |
|--------|------------------------|
| Packet loss rate | > 5% (0.05) |
| Average jitter | > 30ms |
| Round-trip time | > 500ms |
| Bandwidth low event ratio | > 25% |

These thresholds align with Microsoft's Call Quality Dashboard (CQD) definitions for "poor" call streams.

## API Limitations to Understand

```powershell
# Key limitations of the Call Records API in production:

# 1. Data latency — query records from at least 15 minutes ago
$safeStartTime = (Get-Date).AddMinutes(-15)

# 2. Rate limiting — 1 request/second for call records
# Add sleep between paginated requests
Start-Sleep -Seconds 1

# 3. 60-day lookback maximum
$maxLookback = (Get-Date).AddDays(-60)

# 4. $expand adds significant response payload — use $select to trim
$callsUri = "https://graph.microsoft.com/v1.0/communications/callRecords" +
            "?`$filter=startDateTime ge $since" +
            "&`$expand=sessions(`$expand=segments(`$select=media,caller,callee))" +
            "&`$select=id,startDateTime,endDateTime,type,participants"

# 5. Beta-only: detailed per-participant IP and subnet data
# v1.0 network object has limited fields vs beta
```

For high-volume tenants (many calls per day), consider querying by specific date ranges rather than pulling all records at once, and store results incrementally in a database rather than re-querying the full 60-day window on every report run.

## Summary

- **Reports API:** `getTeamsDeviceUsageDistributionUserCounts` provides aggregate device-type usage counts; returns CSV that must be parsed with `ConvertFrom-Csv`.
- **Call Records data model:** Call → Sessions → Segments → Media → Streams; audio quality metrics (packet loss, jitter, RTT) live in the `network` object on each stream.
- **Room endpoint identification:** Check the `userAgent.headerValue` on the caller/callee endpoint to identify Teams Rooms devices within call records.
- **Poor call thresholds:** Align with CQD definitions — packet loss > 5%, jitter > 30ms, RTT > 500ms.
- **Limitations:** 15-minute data latency after call end, 60-day lookback maximum, 1 request/second rate limit, large response payloads from `$expand` — use `$select` to trim.
