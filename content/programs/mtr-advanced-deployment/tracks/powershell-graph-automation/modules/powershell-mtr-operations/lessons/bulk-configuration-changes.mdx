## Overview

Bulk configuration changes are among the highest-risk PowerShell operations in an MTR deployment — a misconfigured `foreach` loop touching 200 room accounts at once can create 200 incidents simultaneously. The discipline applied to bulk changes — always running with `-WhatIf` first, logging every operation, processing in batches, and collecting errors without failing silently — is what separates a trusted automation script from one that requires a late-night incident call. This lesson covers the four most common bulk change scenarios and the safe pattern that wraps all of them.

---

## The Safe Bulk Change Pattern

Every bulk change script should follow this pattern regardless of what operation is being performed:

```powershell
# STEP 1: Define the change scope
$targetRooms = Get-Mailbox -RecipientTypeDetails RoomMailbox -ResultSize Unlimited |
    Where-Object { $_.CustomAttribute1 -eq "LON-HQ" }   # Building filter

Write-Host "Target scope: $($targetRooms.Count) rooms"

# STEP 2: WhatIf pass — validate without making changes
Write-Host "`n--- WhatIf pass (no changes made) ---"
foreach ($room in $targetRooms) {
    Write-Host "[WhatIf] Would update $($room.DisplayName) ($($room.UserPrincipalName))"
}

# Pause for operator confirmation
$confirm = Read-Host "`nProceed with $($targetRooms.Count) changes? (yes/no)"
if ($confirm -ne "yes") { Write-Host "Aborted."; exit }

# STEP 3: Execute in batches with logging
$results   = [System.Collections.Generic.List[PSObject]]::new()
$errors    = [System.Collections.Generic.List[PSObject]]::new()
$batchSize = 20
$batchNum  = 0

for ($i = 0; $i -lt $targetRooms.Count; $i += $batchSize) {
    $batchNum++
    $batch = $targetRooms[$i..([Math]::Min($i + $batchSize - 1, $targetRooms.Count - 1))]
    Write-Host "Processing batch $batchNum ($($batch.Count) rooms)..."

    foreach ($room in $batch) {
        try {
            # <--- THE ACTUAL CHANGE GOES HERE --->
            # Set-CalendarProcessing, Grant-CsTeamsMeetingPolicy, etc.

            $results.Add([PSCustomObject]@{
                Room      = $room.DisplayName
                UPN       = $room.UserPrincipalName
                Status    = "Success"
                Timestamp = (Get-Date -Format "o")
            })
        } catch {
            $errors.Add([PSCustomObject]@{
                Room      = $room.DisplayName
                UPN       = $room.UserPrincipalName
                Status    = "Failed"
                Error     = $_.Exception.Message
                Timestamp = (Get-Date -Format "o")
            })
            Write-Warning "Failed: $($room.DisplayName) — $($_.Exception.Message)"
        }
    }

    # Throttle prevention between batches
    if ($i + $batchSize -lt $targetRooms.Count) {
        Write-Host "Batch $batchNum complete. Sleeping 5s to avoid throttling..."
        Start-Sleep -Seconds 5
    }
}

# STEP 4: Summary and log export
$date = Get-Date -Format "yyyy-MM-dd-HHmm"
$results | Export-Csv -Path "C:\Logs\bulk-change-success-$date.csv" -NoTypeInformation
$errors  | Export-Csv -Path "C:\Logs\bulk-change-errors-$date.csv"  -NoTypeInformation

Write-Host "`n--- Summary ---"
Write-Host "Success: $($results.Count)"
Write-Host "Failed:  $($errors.Count)"
if ($errors.Count -gt 0) { Write-Warning "Review errors: C:\Logs\bulk-change-errors-$date.csv" }
```

This shell applies to all four bulk scenarios below — only the operation inside `try {}` changes.

## Bulk Calendar Processing Updates

Updating `BookingWindowInDays` for all rooms in a building so they accept bookings up to 180 days in advance:

```powershell
# Targeting rooms in building LON-HQ (stored in CustomAttribute1)
$targetRooms = Get-Mailbox -RecipientTypeDetails RoomMailbox -ResultSize Unlimited |
    Where-Object { $_.CustomAttribute1 -eq "LON-HQ" }

# Inside the try block of the safe pattern above:
Set-CalendarProcessing -Identity $room.UserPrincipalName `
    -BookingWindowInDays 180 `
    -AutomateProcessing AutoAccept `
    -AllowRecurringMeetings $true `
    -MaximumDurationInMinutes 480
```

Other common bulk `Set-CalendarProcessing` changes:

| Parameter | Typical Bulk Update Scenario |
|-----------|------------------------------|
| `BookingWindowInDays` | Policy change affecting all rooms in a region |
| `AutomateProcessing` | Switching rooms from AutoAccept to None for a maintenance window |
| `AddAdditionalResponse` | Adding a standard footer message to all room acceptance emails |
| `DeleteComments` | Policy enforcement — removing meeting body content for privacy |
| `MaximumDurationInMinutes` | Updating cap for rooms now used for half-day workshops |

## Bulk Meeting Policy Reassignment

Assigning a custom Teams meeting policy to all room accounts in a building:

```powershell
# Get all Teams room user objects in a specific office
$roomUsers = Get-CsOnlineUser -Filter "Department -eq 'LON-HQ-Rooms'" |
    Where-Object { $_.TeamsRoomsLicenseConfiguration -ne $null }

# Inside the try block:
Grant-CsTeamsMeetingPolicy -Identity $room.UserPrincipalName `
    -PolicyName "MTR-LON-MeetingPolicy"
```

<Callout type="tip">
`Get-CsOnlineUser` with a filter on `Department` is reliable for building-level scoping when the Department attribute is populated during provisioning. Alternatively, filter by `EnterpriseVoiceEnabled`, license assignment, or an Entra ID group membership.
</Callout>

## Bulk Entra ID Group Membership

Adding all rooms from a new building to the `MTR-Building-LON` Entra ID security group (used for Intune policy targeting):

```powershell
# Get target group
$group = Get-MgGroup -Filter "displayName eq 'MTR-Building-LON'"
$groupId = $group.Id

# Get all room user objects (Entra ID user objects for resource accounts)
$targetRooms = Get-MgUser -Filter "startswith(displayName,'MTR-LON')" -All |
    Select-Object Id, DisplayName, UserPrincipalName

# Inside the try block:
New-MgGroupMember -GroupId $groupId -DirectoryObjectId $room.Id
```

To remove rooms from a group when a building is decommissioned:

```powershell
# Remove member — uses same safe pattern
Remove-MgGroupMemberByRef -GroupId $groupId -DirectoryObjectId $room.Id
```

## Bulk Custom Attribute Updates

Setting `CustomAttribute1` to a building code for all rooms in a site — useful when onboarding a new building and the provisioning script didn't set the attribute, or when a building code changes:

```powershell
$targetRooms = Get-Mailbox -RecipientTypeDetails RoomMailbox -ResultSize Unlimited |
    Where-Object { $_.UserPrincipalName -like "*-LON-*" }

# Inside the try block:
Set-Mailbox -Identity $room.UserPrincipalName `
    -CustomAttribute1 "LON-HQ" `
    -CustomAttribute2 "UK" `
    -CustomAttribute3 "Standard"
```

Custom attributes are used throughout the fleet inventory and reporting scripts as the primary metadata store for building, region, and room type. Keeping them accurate is critical for all downstream automation.

## Summary

- **Safe bulk pattern:** Always scope → WhatIf → confirm → batch execute with `try/catch` → export logs → summarize. Never skip any step.
- **Batching:** Process rooms in groups of 20 with `Start-Sleep` between batches to avoid Exchange Online and Teams throttling.
- **Error collection:** Use a `List[PSObject]` to accumulate per-operation errors without stopping the loop — a single failure should not abort 149 other changes.
- **Calendar processing:** `Set-CalendarProcessing` is the most common bulk change; scope by `CustomAttribute1` for building-level targeting.
- **Policy and group changes:** `Grant-CsTeamsMeetingPolicy` and `New-MgGroupMember` follow the same safe pattern with only the operation changing inside `try {}`.
