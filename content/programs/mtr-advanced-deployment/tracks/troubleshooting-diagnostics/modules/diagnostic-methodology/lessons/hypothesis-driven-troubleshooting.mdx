## Overview

The most common failure mode in technical troubleshooting is jumping to a solution before understanding the problem. A technician who restarts a device because "that usually fixes it" sometimes gets lucky and resolves the issue; but when the issue recurs, or when the restart doesn't fix it, they are left without a path forward. Hypothesis-driven troubleshooting starts from the evidence and builds a systematic path to root cause — every action is a test of a specific hypothesis, and every result narrows the problem space.

---

## The Diagnostic Framework

The hypothesis-driven methodology has five phases:

**Phase 1 — Define the problem precisely:**
The initial report ("the room doesn't work") is rarely specific enough to diagnose. Before taking any action, establish:
- What exactly is failing? (audio, video, joining meetings, display, specific feature)
- When did it start? (specific time, after an update, after a room configuration change)
- How consistently does it fail? (every meeting, intermittently, only at certain times)
- Who is affected? (one room, one building, all rooms)
- Has anything changed recently? (Teams app update, network change, hardware replacement)

*A specific problem statement is the input to the diagnostic process:* "The camera shows in TAC as healthy, but remote participants see a frozen frame approximately 20 minutes into any meeting started in Room B-204."

**Phase 2 — Gather evidence:**
Collect relevant data before forming a hypothesis. Evidence for MTR issues:
- Pro Management incident history and AI analysis
- TAC device health status and peripheral health
- CQD call quality data for the room
- Windows Event Log on the device (if accessible)
- Teams app log files (collected via Pro Management log collection)
- Network monitoring data during the affected period

**Phase 3 — Form hypotheses:**
Based on the evidence, generate a set of hypotheses ordered by probability:
- H1 (most likely): [explanation of why this is most likely, based on evidence]
- H2 (second most likely): ...
- H3: ...

For the frozen video example:
- H1: USB bandwidth saturation — camera USB bandwidth may be insufficient for 20-minute sessions (USB hub issue, USB 2.0 port limitation)
- H2: Network bandwidth issue — video bitrate degrades after 20 minutes due to network congestion at that time of day
- H3: Camera overheating — camera firmware issue causes thermal throttling after extended use
- H4: Teams app memory leak — app accumulates memory over 20 minutes until video subsystem is affected

**Phase 4 — Design tests, cheapest first:**
For each hypothesis, design a test that would either confirm or eliminate it. Order tests by cost (time, disruption, complexity) — start with the cheapest test that would eliminate the most hypotheses.

For the hypotheses above:
- Test H2 first (cheapest): Check CQD for the affected calls — does packet loss or bandwidth data show degradation at the 20-minute mark?
- Test H1 next: Connect the camera directly to the compute unit (bypass USB hub) — does the freeze still occur?
- Test H3: Check camera model for known firmware issues; check device temperature via thermal imaging or device log
- Test H4: Collect app memory profile during a call via Pro Management log collection

**Phase 5 — Document and close:**
Once root cause is confirmed:
- Document the problem, evidence gathered, hypotheses formed, tests run, and root cause found
- Document the resolution and any steps taken to prevent recurrence
- Update the team's diagnostic runbook with this pattern if it is new

## Symptoms vs. Root Causes

A critical diagnostic skill is distinguishing between symptoms (what the user sees) and root causes (why it's happening). Acting on a symptom without understanding the root cause leads to incomplete fixes.

**Common symptom/root cause pairs in MTR:**

| Symptom | Could Be Root Cause |
|---------|-------------------|
| Far end hears echo | AEC not configured; AEC reference signal not connected; speaker too close to microphone |
| Device shows Offline in TAC | Network connectivity lost; device powered off; Teams app not running; resource account sign-out |
| Video freezes periodically | Packet loss (network); USB camera issue; camera driver crash; Teams app bug |
| Meeting quality is consistently poor in one room | Room network issue; QoS not applied; microphone coverage problem; acoustic issue |
| Resource account keeps signing out | NTP out of sync; CA policy blocking; MFA not excluded; password expired |
| Device shows Critical in Pro Management | Camera not detected; Teams app not responding; authentication failure |

The same symptom can have different root causes — diagnosing the symptom correctly requires evidence, not assumption.

---

## Building a Diagnostic Runbook Entry

Each novel MTR issue investigated and resolved should be documented as a runbook entry for the support team. A runbook entry format:

```
ISSUE: [Clear problem statement]
SYMPTOMS: [What users/monitoring sees]
EVIDENCE: [What data pointed to the root cause]
ROOT CAUSE: [Confirmed cause]
RESOLUTION: [Steps taken to fix]
VERIFICATION: [How to confirm resolution]
RECURRENCE PREVENTION: [Configuration change or monitoring added]
TIME TO RESOLVE: [From detection to resolution]
RELATED TO: [Similar issues or known patterns]
```

Over time, a populated runbook allows the support team to identify a familiar pattern within seconds of receiving a new incident — dramatically reducing MTTR.

---

## When Not to Solve Immediately

Not every MTR issue should be resolved by the first person who sees it. Escalation indicators:

- Issue recurs after the same "fix" has been applied twice — pattern suggests misdiagnosed root cause
- Multiple rooms exhibiting the same symptom simultaneously — likely a platform-level change (Microsoft 365 service incident, network change, policy update)
- Issue is outside the technical scope of the current tier (Teams service incident requires Microsoft escalation)
- Issue has compliance or security implications (unauthorized sign-in, unusual access pattern)

In these cases, escalate according to the support structure rather than continuing to apply incorrect fixes.

## Summary

- **Five phases:** Define precisely → Gather evidence → Form hypotheses → Test cheapest first → Document.
- **Symptoms vs. root causes:** Never treat the symptom description as the cause; ask "why" at each level.
- **Hypothesis ordering:** Most likely first; test cheapest tests first; each test eliminates one or more hypotheses.
- **Runbook:** Document every novel issue; a populated runbook converts investigation time to pattern recognition time.
- **Escalation triggers:** Same fix applied twice without resolution; fleet-wide symptoms; security implications.
