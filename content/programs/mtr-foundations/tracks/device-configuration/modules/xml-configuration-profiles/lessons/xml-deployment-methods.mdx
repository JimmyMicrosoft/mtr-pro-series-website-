## Overview

A correctly written SkypeSettings.xml file has no effect until it reaches the target device in the right location. The deployment method determines how the file gets there — and for fleet deployments, the difference between a manual method and a scripted method is the difference between touching every device and touching none of them. This lesson covers all four supported deployment methods, when to use each, and the precedence rules that determine which configuration source wins when multiple methods are in use simultaneously.

---

## Method 1: Manual File Copy

The simplest deployment method — copy the XML file directly to the LocalState folder on the device.

**Target path:**
```
C:\Users\Skype\AppData\Local\Packages\Microsoft.SkypeRoomSystem_8wekyb3d8bbwe\LocalState\SkypeSettings.xml
```

**Procedure:**

1. Connect a keyboard and mouse to the MTR Windows device (or use Remote Desktop if enabled)
2. Sign in with the local admin account (not the Skype/resource account)
3. Navigate to the target path using File Explorer (you may need to enable "Show hidden files and folders" in View options to see the AppData directory)
4. Copy or overwrite the SkypeSettings.xml file at that location
5. Reboot the device — the Teams app reads the file at startup

**When to use:** Single-device deployments, testing during commissioning before fleet deployment, one-off configurations for individual rooms with unique requirements.

**Limitations:** Does not scale beyond a handful of devices. Requires physical or remote access to each device. Not repeatable or auditable at fleet scale.

<Callout type="tip">
  During commissioning, always verify the XML configuration using the manual
  method on the first device before deploying to the full fleet. Confirm the
  expected behavior, then scale with USB or Intune for the remaining devices.
</Callout>

## Method 2: USB Drive Deployment

MTR Windows devices can read a SkypeSettings.xml file from a USB drive at startup. This is the fastest method for on-site deployments across multiple devices without network-based management.

**USB drive requirements:**
- File system: FAT32 (not NTFS, not exFAT — FAT32 is required)
- The XML file must be at the root of the drive: `\SkypeSettings.xml`
- Only the XML file is required; no other structure is needed

**Procedure:**

1. Format a USB drive as FAT32
2. Copy the validated SkypeSettings.xml to the root of the drive (not in a folder)
3. Insert the USB drive into the MTR Windows device while it is running
4. Reboot the device — on startup, the Teams app detects the USB drive, copies the SkypeSettings.xml to the LocalState folder, and applies the configuration
5. Remove the USB drive after the device has fully booted and the configuration has been applied

**Verification:** After reboot, navigate to the LocalState folder and confirm the SkypeSettings.xml file is present and contains the expected content.

**When to use:** On-site deployments of 5–30+ devices, rapid configuration of devices before they go online for network management, deployments where network-based management is not yet configured.

**Limitations:** Requires physical presence at each device to insert the USB drive. Does not provide ongoing management — if the XML needs to change later, the USB process must be repeated.

## Method 3: Intune PowerShell Script

For organizations using Microsoft Intune for device management, a PowerShell script can place the XML file at the correct path on any Intune-enrolled MTR Windows device without requiring physical access.

**Script approach:**

The script performs three actions:
1. Defines the XML content as a string variable
2. Constructs the correct target path based on the device's Skype user profile
3. Writes the file to that path

```powershell
# MTR XML Deployment Script — Intune
# Run context: SYSTEM account

$xmlContent = @"
<?xml version="1.0" encoding="UTF-8"?>
<SkypeSettings>
  <DualMonitorMode>true</DualMonitorMode>
  <ContentDuplicationAllowed>false</ContentDuplicationAllowed>
  <AutoScreenShare>true</AutoScreenShare>
  <FrontRowEnabled>true</FrontRowEnabled>
</SkypeSettings>
"@

# Build target path using the Skype user's SID-based package path
$packageBase = "C:\Users\Skype\AppData\Local\Packages"
$packageFolder = Get-ChildItem $packageBase -Filter "Microsoft.SkypeRoomSystem_*" |
    Select-Object -First 1 -ExpandProperty Name

if ($packageFolder) {
    $targetPath = Join-Path $packageBase "$packageFolder\LocalState\SkypeSettings.xml"
    $xmlContent | Out-File -FilePath $targetPath -Encoding UTF8 -Force
    Write-Output "SkypeSettings.xml deployed to: $targetPath"
} else {
    Write-Error "SkypeRoomSystem package directory not found."
    exit 1
}
```

**Key script design notes:**

- The script uses `Get-ChildItem` with a wildcard to find the `Microsoft.SkypeRoomSystem_*` folder rather than hardcoding the full package path. The suffix after the underscore is a publisher hash that may differ across Windows versions and should not be hardcoded.
- The script runs under the SYSTEM account in Intune — it has access to all user profiles including the Skype user's AppData.
- After the script runs, the device must reboot for the Teams app to read the new XML. This can be combined with an Intune device reboot policy or scheduled during a maintenance window.

**Intune deployment configuration:**

- Platform: Windows 10 and later
- Script type: PowerShell script
- Run this script using the logged on credentials: **No** (run as SYSTEM)
- Enforce script signature check: organization-dependent
- Assign to the device group containing MTR Windows devices

**When to use:** Fleet deployments where devices are Intune-enrolled. The most scalable method for organizations already using Intune. Changes can be pushed to all devices simultaneously by updating the script assignment.

## Method 4: Teams Admin Center Configuration Profiles

Teams Admin Center (TAC) provides a configuration profile feature that delivers settings to MTR devices without requiring any file-level management. This is the Microsoft-native, Teams-integrated approach to device configuration.

**What TAC config profiles cover:**

TAC configuration profiles control a subset of the settings available in SkypeSettings.xml, delivered through the Teams management layer rather than the XML file layer. Settings available in TAC profiles include display behavior, meeting settings, and some peripheral assignments — but not all SkypeSettings.xml elements.

**Creating and assigning a TAC config profile:**

1. In Teams Admin Center, navigate to **Teams Devices → Rooms on Windows**
2. Select **Configuration profiles** from the left navigation
3. Create a new profile — configure the desired settings
4. Assign the profile to individual devices or to a device tag (group)
5. The profile is delivered to devices over the Teams service connection — no physical access required

**License requirement:** TAC configuration profiles require a **Teams Rooms Pro** license. Devices with a Teams Rooms Basic license cannot receive TAC configuration profiles.

**When to use:** Organizations with TAC management capability and Pro licenses. The cleanest management experience for ongoing configuration changes — profiles can be updated in TAC and pushed to all assigned devices without any scripting.

## Configuration Precedence Order

When multiple deployment methods are in use simultaneously — which is common in larger organizations — the following precedence order determines which configuration takes effect:

```
TAC Configuration Profile  (highest priority — overrides everything below)
         ↓
Intune-deployed SkypeSettings.xml
         ↓
Local SkypeSettings.xml (manual or USB deployed)
         (lowest priority)
```

**Practical implications:**

- If a TAC configuration profile is assigned to a device, its settings override the corresponding settings in any local SkypeSettings.xml file. Settings controlled by the TAC profile cannot be overridden by local XML.
- If Intune deploys an XML file and a local XML file also exists, Intune's file overwrites the local file (since the script deploys to the same path with `-Force`).
- Settings not controlled by a higher-priority source fall through to the lower-priority source. A TAC profile that configures display settings but does not configure content camera settings allows the local XML's content camera settings to remain in effect.

**Troubleshooting implication:** When a local XML configuration doesn't appear to be taking effect, always check whether a TAC profile is assigned to the device. TAC → Devices → select device → Configuration profile tab. If a profile is assigned, its settings supersede the local XML for any overlapping settings.

## Summary

By completing this lesson, you can now:

- **Manual copy:** Direct file placement to the LocalState path — correct for single-device commissioning and testing.
- **USB deployment:** FAT32 drive, XML at root — fastest on-site method for pre-Intune fleet deployment.
- **Intune script:** Dynamic path resolution script, run as SYSTEM, assigns to MTR device group — scalable fleet method for Intune-enrolled organizations.
- **TAC config profiles:** Pro license required, delivered over Teams service connection, no file management needed — cleanest ongoing management approach.
- **Precedence:** TAC Profile > Intune XML > Local XML. TAC profile settings override all lower-priority sources for settings they control.
