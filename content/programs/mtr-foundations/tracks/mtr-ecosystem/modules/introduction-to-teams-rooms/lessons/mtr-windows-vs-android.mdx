## Overview

At some point in nearly every MTR project, someone asks the question: Windows or Android? It might come from a client who just saw a sleek bar-style device at a trade show. It might come from a project manager who found a lower-cost Android option and wants to know if it is "just as good." It might come from an IT administrator who has a preference based on their existing management infrastructure. Whatever the context, your answer will shape the feature set, the installation approach, the management workflow, and the client's long-term satisfaction with the room.

This is not a question with a universal right answer. Both platforms run the Microsoft Teams Rooms application. Both join meetings, display calendars, route audio and video through certified peripherals, and appear in the Teams Admin Center. But beneath that surface similarity, the two platforms have meaningful architectural differences that determine which features are available, how the system is updated, and how flexible the hardware choices are. Getting this wrong — specifying Android hardware for a room that needs Intelligent Speaker, or overbuilding a four-seat huddle space with a full Windows compute stack — costs money and credibility.

By the end of this lesson you will be able to articulate those differences clearly, identify the feature areas where the platforms diverge, and make a confident platform recommendation for any room scenario you encounter.

## MTR on Windows — Architecture and Capabilities

<GlossaryTerm slug="mtr-windows">MTR Windows</GlossaryTerm> runs on Windows 10 or Windows 11 IoT Enterprise, a specialized edition of Windows designed for fixed-function kiosk deployments. When a Windows-based MTR compute unit boots, it launches directly into the Teams Rooms application — there is no standard Windows desktop, no Start menu, no ability for a room user to open a browser or run other applications. The OS is locked into a kiosk shell that exposes only the Teams Rooms interface. This is not a limitation; it is the intentional design. The room is a meeting endpoint, not a general-purpose computer.

The compute unit is a discrete hardware component — a small form-factor PC that mounts in an AV rack, behind a display, or inside AV furniture. It connects to all room peripherals via USB 3.0 (camera, audio device, touch console in most configurations) and to displays via HDMI or DisplayPort. Some configurations use IP-based touch consoles (like the Logitech Tap IP or the Crestron Flex panel), which connect to the compute unit via Ethernet rather than USB, extending cable run lengths significantly and simplifying installation in rooms with greater distances between the table and the rack.

The Windows platform supports the full Teams Rooms feature set. Intelligent Speaker, Front Row layout, Coordinated Meetings, dual display with content on second screen, content camera AI, NDI output, deep XML configuration — if Microsoft has shipped a feature for Teams Rooms, it is available on Windows first and in its most complete form. This is because Microsoft's engineering teams develop primarily against the Windows platform, and Android support is added subsequently as OEM firmware matures to support new capabilities.

Representative Windows-based compute solutions include the Lenovo ThinkSmart Core, the Crestron UC-MX70-T, the HP Presence Compute Module, and Intel NUC-based bundles paired with controllers like the Logitech Tap IP. These units vary in processing power, port count, and form factor, but all run the same Teams Rooms application stack.

<Callout type="note">
  The kiosk lockdown on MTR Windows means the IT team cannot use the device as a general-purpose
  workstation. This is by design and should be communicated clearly to clients who ask whether
  they can "also use the Teams Rooms PC for presentations from a laptop." The answer is no —
  the device is dedicated to the Teams Rooms function. Laptop content is shared via HDMI input
  to the room system, not by running software on the compute unit.
</Callout>

## MTR on Android — Architecture and Form Factors

<GlossaryTerm slug="mtr-android">MTR Android</GlossaryTerm> runs on a custom Android AOSP (Android Open Source Project) build created by each device OEM. Unlike the Windows platform — where the OS is standardized across manufacturers — on Android, Poly ships its own firmware, Yealink ships its own, Logitech ships its own, and so on. The Teams Rooms application itself is the same Microsoft application across all Android OEMs, but the underlying firmware layer that supports it is manufacturer-specific.

The defining characteristic of Android MTR hardware is the all-in-one form factor. Rather than a separate compute unit, display, camera, and audio device, Android devices integrate most or all of these functions into a single bar or panel. The Yealink MeetingBar A40 mounts above or below a display and contains an integrated camera, microphone array, and speakers — all in one device powered by a single connection. The Poly Studio X52 does the same. The Neat Bar, the DTEN D7X, and similar devices follow the same concept. For small rooms and huddle spaces, this simplicity is a genuine advantage: fewer mounting points, fewer cables, faster installation, and less for the IT team to manage.

Because the compute function is integrated into the device, there is no separate compute unit to spec or rack-mount for an Android deployment. The hardware selection is the device itself. This is an important distinction when building BOMs and writing SOWs — the installation scope for an Android room is structurally simpler than a Windows room.

The tradeoff for this simplicity is that Android hardware has historically lagged the Windows platform on feature parity. The reasons are architectural: new Teams Rooms features often require OS-level capabilities that must be built into OEM firmware, and each OEM ships firmware updates on their own schedule. When Microsoft releases a new feature for Teams Rooms, Windows devices receive it through the Microsoft Store and Windows Update relatively quickly. Android devices receive it when their OEM ships an updated firmware build — which may be weeks or months later.

<Callout type="tip">
  Before committing to Android hardware for any project, check the current Microsoft Teams Rooms
  feature comparison matrix at learn.microsoft.com. The gap between Windows and Android narrows
  with each app release, but specific features — particularly AI-driven capabilities — may not
  yet be available on Android. Never rely on documentation that is more than a few months old;
  feature availability changes frequently.
</Callout>

## The Feature Parity Gap

The feature parity gap between Windows and Android is not a flaw in Microsoft's product strategy — it is a natural consequence of how the platforms are structured. What matters for integrators is knowing which gaps are material to a given project and communicating them accurately to clients. Here are the feature areas where the two platforms differ most significantly at the current product release:

**Intelligent Speaker** is available on MTR Windows only. This feature, which identifies individual meeting participants by voice using a Teams Rooms Pro license and certain certified microphone arrays, is not available on Android devices. It is one of the most requested features for executive boardrooms and premium deployments — confirm platform before specifying.

**Front Row layout** — which places remote participant video in a horizontal strip at the bottom of the display to create a more natural eye-contact experience — is supported fully on Windows and partially on select Android devices. Not all Android hardware supports it and behavior can vary across OEMs.

**Dual display** with independent content and people feeds is fully supported on Windows (up to two displays, configurable via XML). Android support for dual display is improving but varies significantly across devices — some Android hardware supports it, some does not, and behavior differs from the Windows implementation.

**Content camera AI**, which digitizes and enhances whiteboard content and optionally removes the presenter standing in front of the whiteboard, is a Windows-only feature. Android does not support content camera AI processing.

**Coordinated Meetings**, which allows MTR rooms and Surface Hub devices in the same physical space to join a meeting together with shared audio management, is a Windows feature with no equivalent on Android.

**XML configuration depth** is substantially greater on Windows. The SkypeSettings.xml schema on Windows covers dozens of configuration parameters including display layout, meeting defaults, theming, and peripheral selection. Android devices support a more limited XML subset, and some configuration must be done through the Teams Admin Center or the device's OEM management interface.

**NDI output**, which streams room audio and video over the network for integration with broadcast or recording tools, is available on Windows and not on Android.

## Update Control and Peripheral Flexibility

Beyond features, the two platforms differ in how they receive updates and in how flexible the peripheral ecosystem is — both of which have practical consequences for integrators and IT teams.

On Windows, Teams Rooms app updates are delivered through the Microsoft Store, and Windows OS updates come through Windows Update. IT administrators who manage devices with Intune or Windows Update for Business can defer updates, test them in a pilot group before broad deployment, and roll back if a problematic release affects room functionality. This degree of control is important for organizations that operate in regulated industries or that have experienced issues with automatic updates disrupting room behavior. The update pipeline is predictable and manageable.

On Android, the Teams Rooms application updates on a similar cadence, but the firmware layer is controlled by the OEM. A new firmware build from Yealink or Poly may be required to support features in a new app release, and that firmware ships on the OEM's schedule. In practice, this can create a situation where the Teams Rooms app on an Android device references a capability that the current OEM firmware does not yet support. The result is a feature that appears to be configured but does not work — which generates support calls and client confusion. Integrators should communicate this update model honestly to clients who are evaluating Android hardware.

On the peripheral side, the Windows platform's modularity is a significant advantage. Because all peripherals connect via USB or IP, and because Microsoft maintains a broad certified peripherals catalog, integrators can mix and match cameras, audio devices, and controllers from different manufacturers on the same Windows compute unit. Want a Shure ceiling microphone array with a Logitech camera and a Crestron touch console? Provided all three are certified for MTR Windows, that combination works. This flexibility is valuable when rooms have specific audio or visual requirements that no single vendor's bundle satisfies.

Android devices are more constrained. Most all-in-one Android devices have integrated cameras and microphones designed to work as a unit. Adding external peripherals is possible on some devices — for example, connecting a PTZ camera via USB to a MeetingBar — but the OEM determines what is certified for their device, and the options are narrower than the Windows catalog. For rooms where the integrated camera and audio bar meets the requirement, this is fine. For rooms with specific peripheral requirements, Windows gives you more options.

## Choosing the Right Platform

Given everything above, a practical framework for platform selection looks like this. Choose MTR Windows when: the room is medium to large and the integrated camera and audio of an Android bar would not provide adequate coverage; when the client requires Intelligent Speaker, Front Row, or content camera AI; when dual display with independent feeds is needed; when the IT team wants Intune-managed update deferral; or when deep XML configuration is needed for custom room behavior.

Choose MTR Android when: the room is small (up to six seats) and an all-in-one bar form factor is appropriate; when installation simplicity and a minimal device footprint are priorities; when the required feature set is core meeting join, one-touch join, audio and video, and content sharing — and the Android platform delivers all of those reliably; or when the client's IT team prefers fewer managed endpoints (one Android device vs. a compute unit plus multiple peripherals).

{/* MEDIA: Room size and platform selection matrix — visual table showing small (up to 6 seats), medium (6–12 seats), and large (12+ seats) room profiles with recommended platform, typical device example, and key decision factors for each. Filename: p1-t1-m1-l2-platform-selection-matrix.png */}

The key point is this: Android is not inferior to Windows — it is optimized for a different use case. An Android bar in a four-seat huddle room is an excellent solution. The same Android bar in a fourteen-seat boardroom where the client expects Intelligent Speaker and a 180-degree dual-camera view is the wrong specification. Match the platform to the requirement.

## Summary

By completing this lesson, you can now:

- **Describe the MTR Windows architecture:** A Windows IoT Enterprise kiosk shell running the Teams Rooms application on a discrete compute unit connected to USB and IP peripherals. Supports the full Teams Rooms feature set and allows flexible mix-and-match peripheral selection from the certified catalog.
- **Describe the MTR Android architecture:** An OEM-specific AOSP firmware layer running the Teams Rooms Android application, typically in an all-in-one form factor. Simpler to install, smaller footprint, but historically trails Windows on feature parity and has less flexible peripheral options.
- **Identify the key feature gaps:** Intelligent Speaker, content camera AI, Coordinated Meetings, full dual display support, NDI output, and deep XML configuration are Windows advantages. Android excels in all-in-one simplicity for small rooms.
- **Apply a platform selection framework:** Match platform to the room's size, feature requirements, and IT management preferences — not to a default assumption that one is better than the other.
