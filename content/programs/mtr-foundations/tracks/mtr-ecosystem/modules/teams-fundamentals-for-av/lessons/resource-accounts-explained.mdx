## Overview

A resource account is the identity that an MTR device signs into, and it is more complex than it appears. Under the hood, a resource account is two things simultaneously: an Entra ID (formerly Azure AD) user object that holds the Teams license and authentication identity, and an Exchange Online room mailbox that handles meeting invites. Both halves must be correctly configured before the device will function. When one half is misconfigured, the device may sign in successfully but still not show meetings, fail to join calls, or appear offline in the Teams Admin Center — leaving integrators chasing hardware problems that are actually account problems.

This lesson covers what a resource account actually is at the platform level, the five most common misconfigurations encountered in real deployments, and how to verify account health using PowerShell before you ever touch the device. The PowerShell commands here are the same ones used by Microsoft support engineers when diagnosing MTR deployment issues — knowing them puts you in a position to diagnose problems and communicate precisely with IT.

## What a Resource Account Actually Is

When IT creates a resource account for an MTR room, two objects are created in the Microsoft 365 environment. The first is an Entra ID user account — the same type of object as any employee account — with an assigned Teams Rooms license. This is the identity the MTR device uses to authenticate with Microsoft's cloud services, the account that appears in Teams Admin Center under Devices, and the identity that Teams uses to identify which room is in a meeting.

The second object is an Exchange Online room mailbox. This is what enables calendar-based meeting join. When a meeting organizer invites the room, Exchange Online receives the invite at the room mailbox and — if AutomateProcessing is correctly configured — automatically accepts it. The accepted invite then appears in the mailbox calendar, which the MTR device reads to display upcoming meetings on the touch console.

Both objects share the same UPN (user principal name — the account's email address) and are linked at creation. But they can be misconfigured independently. An integrator who sees that the Entra ID account is licensed correctly but meetings are not appearing on the console should immediately look at the Exchange Online room mailbox configuration — the two layers have separate settings and separate failure modes.

## The Five Common Misconfigurations

These five issues account for the overwhelming majority of resource account problems encountered during MTR commissioning. Each one is specific, verifiable, and correctable — but only if you know what to look for.

**1. AutomateProcessing not set to AutoAccept**

This is the single most common cause of rooms not showing calendar meetings. Exchange Online room mailboxes default to AutomateProcessing set to AutoUpdate (or sometimes AutoNone, depending on how the tenant was provisioned). AutoUpdate processes invites but does not automatically accept them. The mailbox must be set to AutoAccept for incoming meeting invites to be automatically accepted and added to the calendar.

The PowerShell command to check this:

```powershell
Get-CalendarProcessing -Identity "room@contoso.com" | Select AutomateProcessing
```

The correct value is `AutoAccept`. If it shows `AutoUpdate` or `AutoNone`, the IT administrator needs to run:

```powershell
Set-CalendarProcessing -Identity "room@contoso.com" -AutomateProcessing AutoAccept
```

After setting AutoAccept, future meeting invites will be automatically accepted. Meetings already sent to the room mailbox before this change may need to be resent to appear on the console.

**2. License not assigned or wrong license type**

An MTR device will sign in with the resource account credentials but will not function as a Teams Room device without the correct license. The Teams Rooms Basic or Teams Rooms Pro license must be assigned to the Entra ID user object. Without it, the device may appear to sign in but will not display in the TAC Devices section correctly and will not have access to Teams Rooms-specific features.

To verify license assignment in PowerShell:

```powershell
Get-MsolUser -UserPrincipalName "room@contoso.com" | Select Licenses
```

This requires the MSOnline module. Alternatively, the license can be verified in the Microsoft 365 admin center or the TAC Users section. Confirm that one of the assigned licenses is a Teams Rooms SKU — not a standard Teams user license or a Teams Essentials license, neither of which activates the Teams Rooms application correctly.

**3. Account disabled or blocked from sign-in**

Resource accounts can be created with sign-in blocked by default, particularly in organizations with strict provisioning policies. A blocked account will appear to exist in Entra ID but the MTR device will receive an authentication error when attempting to sign in. This is often misdiagnosed as a password issue.

To check sign-in status:

```powershell
Get-MsolUser -UserPrincipalName "room@contoso.com" | Select BlockCredential
```

A value of `True` means the account is blocked from signing in. The IT administrator needs to unblock it:

```powershell
Set-MsolUser -UserPrincipalName "room@contoso.com" -BlockCredential $false
```

**4. Password expiry**

Resource accounts often get missed during routine password expiry enforcement because they are not associated with a named employee. A resource account whose password has expired will cause the MTR device to be kicked out of its signed-in session and display a sign-in prompt on the touch console — often mistaken for a device fault.

Resource account passwords should be set to never expire. To verify and set this:

```powershell
Get-MsolUser -UserPrincipalName "room@contoso.com" | Select PasswordNeverExpires
Set-MsolUser -UserPrincipalName "room@contoso.com" -PasswordNeverExpires $true
```

Confirm with the IT administrator that password expiry policy for the resource account is set to never expire at the tenant level or that the account is excluded from password expiry enforcement.

**5. Multi-Factor Authentication not excluded**

MFA conditional access policies — if applied tenant-wide — will require the resource account to complete an MFA challenge during sign-in. An MTR device cannot complete an MFA challenge because there is no human present to approve it. MFA applied to a resource account will prevent the device from signing in or will result in token refresh failures that knock the device offline every time the authentication token expires.

Resource accounts must be excluded from MFA conditional access policies. This is an IT administrator task — it requires editing the Conditional Access policy in Entra ID to add the resource account or a security group containing room accounts to the exclusion list. To check whether the account is covered by MFA policies requires examining the Conditional Access policies in Entra ID admin center — this is not accessible via a single PowerShell command but the IT administrator can use the "What If" tool in Conditional Access to simulate which policies would apply to the resource account.

<Callout type="warning">
  MFA exclusion for resource accounts is a deliberate security trade-off that the IT
  administrator must authorize and document. Some organizations use dedicated Conditional
  Access policies for room accounts with location-based restrictions (corporate network
  only) as a compensating control. Never advise an IT administrator to simply "turn off MFA"
  — the correct action is a targeted exclusion for room accounts, not a tenant-wide change.
</Callout>

## Verifying the Full Account Configuration

Before signing a device in for the first time, a complete verification of the resource account covers all five potential issues. The following PowerShell sequence (requiring the MSOnline and ExchangeOnlineManagement modules) checks the critical properties:

```powershell
# Connect to Exchange Online
Connect-ExchangeOnline -UserPrincipalName "admin@contoso.com"

# Check calendar processing (AutomateProcessing must be AutoAccept)
Get-CalendarProcessing -Identity "room@contoso.com" | Select AutomateProcessing, AllowConflicts, DeleteComments, DeleteSubject

# Connect to MSOnline
Connect-MsolService

# Check account status and license
Get-MsolUser -UserPrincipalName "room@contoso.com" | Select DisplayName, BlockCredential, PasswordNeverExpires, Licenses, IsLicensed

# Check Teams-specific settings (requires Teams PowerShell module)
Get-CsOnlineUser -Identity "room@contoso.com" | Select DisplayName, TeamsUpgradeEffectiveMode, EnterpriseVoiceEnabled, AssignedPlan
```

The `Get-CsOnlineUser` command also shows the meeting policy assigned to the account (`TeamsMeetingPolicy` property). Cross-reference this with the policy verification covered in the previous lesson.

A resource account ready for device sign-in should show: `AutomateProcessing = AutoAccept`, `BlockCredential = False`, `PasswordNeverExpires = True`, `IsLicensed = True` with a Teams Rooms SKU in the Licenses list, and the `TeamsUpgradeEffectiveMode` set to `TeamsOnly`.

<Callout type="info">
  The MSOnline module (Get-MsolUser, Set-MsolUser) is deprecated by Microsoft but remains
  widely used and functional as of this writing. The replacement is the Microsoft Graph
  PowerShell SDK. Your organization may already use Graph SDK commands — the properties
  verified are the same; only the command syntax differs. If the IT team uses Graph SDK,
  ask for their equivalent verification commands rather than assuming MSOnline is available.
</Callout>

## Calendar Permissions and Meeting Invite Behavior

Two Exchange Online settings beyond AutomateProcessing can affect how meeting invites are processed by the room mailbox.

**DeleteComments and DeleteSubject:** By default, room mailboxes are configured to strip the meeting body text (`DeleteComments = True`) and replace the meeting subject with the organizer's name (`DeleteSubject = False`, `AddOrganizerToSubject = True`). These are calendar privacy settings. For most rooms this is fine. Some clients, however, want the full meeting details preserved on the console — for example, rooms in shared spaces where the agenda details are helpful for ad hoc attendees. These can be adjusted:

```powershell
Set-CalendarProcessing -Identity "room@contoso.com" -DeleteComments $false -DeleteSubject $false -AddOrganizerToSubject $false
```

**AllowConflicts:** Room mailboxes default to not accepting overlapping meetings (`AllowConflicts = False`). This means if two organizers both send a meeting invite for the same time slot, only the first accepted invite will be added to the calendar. The second will be declined. This is correct behavior for a physical room. Do not change this setting unless the client has a specific reason — shared virtual rooms used for overflow scheduling are a rare legitimate exception.

## Summary

By completing this lesson, you can now:

- **Explain the dual nature of a resource account:** An Entra ID user object (identity, licensing) combined with an Exchange Online room mailbox (calendar processing). Both must be correctly configured.
- **Identify the five common misconfigurations:** AutomateProcessing not set to AutoAccept, missing or wrong license, sign-in blocked, password expires, MFA not excluded. Each has a specific symptom and a specific fix.
- **Use PowerShell to verify before you commission:** `Get-CalendarProcessing`, `Get-MsolUser`, and `Get-CsOnlineUser` together cover all critical verification points. Run these before signing in the device.
- **Know what to escalate to IT:** MFA exclusions and Conditional Access changes require IT administrator authority. Your role is to identify the issue, document it precisely, and communicate it — not to make tenant-level policy changes.
