## Overview

Conditional Access is how Entra ID enforces authentication policies — requiring MFA, restricting access to compliant devices, blocking sign-ins from risky locations. These controls are essential for user accounts and should be applied broadly. For MTR resource accounts, however, the same policies that protect user accounts will break device sign-in if applied without careful exemption design. MTR devices cannot complete an interactive MFA challenge — there is no user present to approve a push notification. This lesson builds the Conditional Access policy structure that applies appropriate security controls to MTR accounts without preventing them from signing in.

---

## Why MFA Breaks MTR Sign-In

Teams Rooms sign-in uses OAuth 2.0 device authentication. The sign-in flow:

1. Device presents resource account UPN and password to Entra ID
2. Entra ID evaluates Conditional Access policies for the account
3. If no MFA policy applies — authentication token issued, device signs in
4. If MFA policy applies — Entra ID requires an additional verification step
5. The device displays the sign-in screen waiting for MFA completion
6. No user is present to complete MFA → sign-in times out → device remains unsigned-in

This failure produces a persistent "Sign in" prompt on the Teams home screen with no error message, or a generic "MFA required" error that the room occupant cannot resolve.

## The Exclusion Group Approach

The standard and Microsoft-recommended approach: create a dedicated Entra ID security group for MTR resource accounts, then exclude that group from all MFA Conditional Access policies.

**Step 1 — Create the MTR service accounts group:**

```powershell
# Create a group for MTR resource accounts
New-MgGroup -DisplayName "MTR Resource Accounts" `
            -Description "Teams Rooms resource accounts — excluded from MFA CA policies" `
            -MailEnabled $false `
            -SecurityEnabled $true `
            -MailNickname "mtr-resource-accounts"
```

**Step 2 — Add all room accounts to the group:**

```powershell
$groupId = (Get-MgGroup -Filter "DisplayName eq 'MTR Resource Accounts'").Id
$rooms = Get-Mailbox -RecipientTypeDetails RoomMailbox -ResultSize Unlimited

foreach ($room in $rooms) {
    $user = Get-MgUser -UserId $room.UserPrincipalName -ErrorAction SilentlyContinue
    if ($user) {
        New-MgGroupMember -GroupId $groupId -DirectoryObjectId $user.Id
    }
}
```

**Step 3 — Exclude the group from MFA policies:**

In Entra ID → Security → Conditional Access → for each MFA policy:
1. Click the policy to edit
2. Under **Users → Exclude**, add the "MTR Resource Accounts" group
3. Save the policy

## What Security Controls Can Apply to MTR Accounts

While MFA cannot be applied, other Conditional Access controls are appropriate and recommended for MTR accounts:

**Named Location restriction:** Create a Conditional Access policy that restricts MTR account sign-ins to trusted corporate IP ranges (the organization's internet egress IPs). This prevents the MTR accounts from being used to sign in from external locations — if an account credential is compromised, the attacker cannot sign in from outside the corporate network.

**Policy design:**
- **Target users:** MTR Resource Accounts group
- **Cloud apps:** Microsoft Teams
- **Conditions:** Named Locations → Exclude trusted locations (invert: block from NON-trusted locations)
- **Grant:** Block access (sign-in from outside corporate IP is blocked)

**Session risk policy:** Apply a Conditional Access policy that evaluates sign-in risk (Entra ID Identity Protection) for MTR accounts. While you cannot require MFA in response to risk (it would break the device), you can configure the policy to block high-risk sign-ins or send an alert to the security team when a risky sign-in is detected.

**Password policy:** Ensure the MTR account passwords are strong (complex, 20+ characters) and rotate them on a schedule. The password is stored on the device and not entered by a user — complexity does not cause operational difficulty.

<Callout type="warning">
  The MTR exclusion group must be kept current. Every newly provisioned room
  account must be added to the exclusion group before the device signs in.
  A room account not in the exclusion group will fail sign-in if MFA is
  required by the existing CA policies. Include group membership in the
  provisioning script (as shown in the previous lesson) to ensure this
  is never missed.
</Callout>

## Monitoring MTR Account Sign-Ins

Even without MFA, MTR account sign-ins should be monitored. Use Entra ID sign-in logs to establish a baseline for normal sign-in behavior:

- Expected sign-in IP range: corporate MTR VLAN egress IPs
- Expected sign-in frequency: one sign-in per device per session (not repeated hourly)
- Expected client application: Teams Rooms on Windows / Teams Rooms on Android

Anomalies to alert on:
- Sign-in from unexpected IP address
- Sign-in using a non-Teams Rooms client application (indicates credential use outside the room device)
- Multiple failed sign-in attempts

Configure Entra ID Identity Protection or a SIEM alert for these patterns.

## Summary

- **MFA exclusion required:** MTR devices cannot complete interactive MFA — exclude room accounts from all MFA CA policies via a dedicated security group.
- **Applicable controls:** Named location restriction (corporate IP only), sign-in risk policy (block high-risk, alert on anomalies), strong passwords on rotation schedule.
- **Keep group current:** Add new accounts to exclusion group during provisioning — include in provisioning script.
- **Monitor sign-ins:** Establish baseline; alert on unexpected IPs, client applications, or high-frequency sign-in attempts.
