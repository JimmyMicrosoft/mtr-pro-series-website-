## Overview

MTR devices are cloud-connected appliances. They authenticate through Entra ID, receive meeting invitations through Exchange Online, join meetings through the Teams media infrastructure, receive updates through the Microsoft Store and OEM cloud portals, and send telemetry to Microsoft. Every one of these functions requires specific network connectivity — and an enterprise firewall that blocks any of them will produce symptoms that look like software bugs or configuration problems, not network blocks. This lesson provides the complete picture of what MTR devices need to reach and how to configure the network to permit that traffic reliably.

---

## Microsoft's Endpoint Category System

Microsoft publishes all M365 network connectivity requirements in a structured endpoint list at `aka.ms/o365endpoints`. The endpoints are organized into three categories:

**Optimize:** The highest-priority traffic — real-time media (audio, video, screen sharing). These endpoints must be reachable with low latency, must not be proxied, and must not be subject to SSL/TLS inspection. Blocking or proxying Optimize traffic directly degrades meeting quality and causes authentication failures.

**Allow:** Important connectivity for Teams sign-in, authentication, CDN access, and telemetry. Should be permitted through the firewall. Proxying is tolerated but adds latency. SSL inspection can break some Allow endpoints.

**Default:** Standard Microsoft 365 services — shared with all M365 products. Treat the same as general internet traffic.

For MTR deployments, the focus is on the **Optimize** and **Allow** categories for Teams and Exchange Online. The Default category can be handled by the organization's standard internet access policy.

## Key URLs and Ports for MTR

The following represent the most critical connectivity requirements for MTR devices. Always verify against the current Microsoft endpoint list, as these are subject to change with Microsoft service updates.

**Teams media (Optimize category):**

| Protocol | Port | Destination | Purpose |
|----------|------|-------------|---------|
| UDP | 3478–3481 | 13.107.64.0/18, 52.112.0.0/14, 52.122.0.0/15 | Teams audio/video media transport (STUN/TURN) |
| TCP | 443 | *.teams.microsoft.com | Teams signaling and fallback media |
| TCP | 443 | *.skype.com | Skype infrastructure (Teams uses Skype transport layer) |

**Authentication and sign-in (Allow category):**

| Protocol | Port | Destination | Purpose |
|----------|------|-------------|---------|
| TCP | 443 | login.microsoftonline.com | Entra ID OAuth authentication |
| TCP | 443 | *.microsoftonline.com | Microsoft identity platform |
| TCP | 443 | *.office.com, *.office.net | Microsoft 365 services |
| TCP | 443 | *.outlook.com | Exchange Online (calendar) |

**Teams application and updates:**

| Protocol | Port | Destination | Purpose |
|----------|------|-------------|---------|
| TCP | 443 | *.azure.net, *.azureedge.net | Teams app CDN delivery |
| TCP | 443 | statics.teams.cdn.office.net | Teams client assets |
| TCP/UDP | 443/80 | *.sfbassets.com | Teams Rooms app assets |

**OEM cloud management portals (varies by OEM):**

| OEM | Portal | Protocol/Port |
|-----|--------|---------------|
| Yealink (YMCS) | ymcs.yealink.com | TCP 443 |
| Poly (Poly Lens) | lens.poly.com | TCP 443 |

| Protocol | Port | Destination |
|----------|------|-------------|

## The Proxy and SSL Inspection Problem

This is where most MTR network issues originate. Enterprise networks commonly deploy:
- **Explicit or transparent HTTP proxies** — all HTTPS traffic is routed through a proxy appliance
- **SSL/TLS inspection (also called deep packet inspection or HTTPS inspection)** — the proxy terminates and re-encrypts TLS connections to inspect content

Both of these are incompatible with Teams Optimize-category traffic for technical reasons:

**Proxy incompatibility with real-time media:** Teams UDP media transport (ports 3478–3481) cannot be proxied — proxies are TCP-only. If a proxy blocks UDP, Teams falls back to TCP/443 media, which increases latency and degrades call quality significantly. Even TCP-based Teams media performs worse through a proxy because the additional hop adds latency that accumulates during real-time communication.

**SSL inspection incompatibility:** Teams uses certificate pinning and custom TLS extensions for authentication tokens. An SSL inspection proxy substitutes its own certificate, which breaks the certificate chain that Teams relies on for authentication. The result is sign-in failures or intermittent authentication token rejection — symptoms that look exactly like resource account configuration problems.

**The correct network configuration:** Configure the firewall and proxy to bypass both proxy routing and SSL inspection for all Microsoft Optimize-category endpoints (the Teams IP ranges and *.teams.microsoft.com, *.skype.com domains). This is sometimes called "proxy bypass" or "direct route" configuration. The Allow-category endpoints should also be excluded from SSL inspection.

<Callout type="warning">
  Never enable SSL inspection on Microsoft Teams media endpoints. It is not
  a security configuration — it is a misconfiguration that causes sign-in
  failures and meeting quality degradation, and it creates a false sense of
  security since Teams traffic is end-to-end encrypted regardless of whether
  the proxy can read it.
</Callout>

## Building the Firewall Rule Set

**Rule construction approach:**

1. Download the current M365 endpoint list from `aka.ms/o365endpoints` — Microsoft provides this as a JSON feed, a CSV, and a web interface
2. Filter for the Teams and Exchange Online service families
3. Build explicit permit rules for all Optimize and Allow endpoints (URL-based rules for FQDNs, IP-based rules for the Teams media IP ranges)
4. Apply proxy bypass configuration for the Optimize category
5. Apply SSL inspection bypass for Optimize and Allow categories
6. Test connectivity from the MTR VLAN using the Microsoft 365 Network Connectivity Test tool (`connectivity.office.com`)

**Microsoft 365 IP Change Management:** Microsoft updates the endpoint list regularly — typically with several weeks' notice published via RSS feed and API. Subscribe to the Microsoft 365 IP Address and URL change notification service to receive advance notice of changes that require firewall rule updates. Failing to track these updates results in blocked connectivity when Microsoft moves to a new IP range.

## Summary

By completing this lesson, you can now:

- **Navigate the endpoint categories:** Optimize (real-time media — must not be proxied or SSL-inspected), Allow (auth and services — should be permitted), Default (standard internet treatment).
- **Build the firewall rule set:** UDP 3478–3481 to Teams media IPs + TCP 443 to Teams/M365 domains. NTP UDP 123. OEM portal TCP 443.
- **Configure proxy and SSL bypass:** Bypass both for all Optimize-category endpoints. SSL bypass for Allow category. Failure to do so causes media degradation and authentication failures.
- **Track endpoint changes:** Subscribe to Microsoft's M365 endpoint change notification to keep firewall rules current.
