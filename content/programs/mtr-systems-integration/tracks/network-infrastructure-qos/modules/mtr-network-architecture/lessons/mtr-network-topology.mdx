## Overview

The network design decisions made before an MTR deployment begins determine how reliably those rooms will perform for the next several years. MTR devices that share a flat network with general office workstations compete for bandwidth with user traffic, expose the room devices to unnecessary broadcast traffic, and cannot be managed independently from the broader network. Proper network segmentation is not optional — it is the baseline architectural decision that enables QoS, simplifies firewall policy, and creates a manageable, supportable device estate.

---

## Why Segment MTR Devices

Three operational reasons drive MTR VLAN segmentation:

**QoS enforcement:** DSCP QoS marking is most reliably applied when MTR devices are on a known subnet. A dedicated MTR VLAN allows the network team to apply QoS policies based on source IP range — ensuring that all traffic from the MTR subnet receives correct DSCP marking regardless of whether the endpoint itself marks traffic correctly.

**Firewall policy scope:** Microsoft publishes a specific set of URLs, IP addresses, and ports required for Teams Rooms connectivity. Applying these as firewall rules is far cleaner when the source is a defined subnet rather than an IP list that changes with every device addition or change.

**Simplified management and monitoring:** Network monitoring tools, DHCP reservations, and access control policies are all simpler to manage when MTR devices live on a known, bounded network segment. A new device added to the MTR VLAN automatically inherits the correct policies.

---

## VLAN Architecture

**Recommended segmentation approach:**

For most enterprise deployments, a single dedicated VLAN for all MTR devices across the site is the standard approach. Some organizations with specific security requirements may segment further — separate VLANs per building or floor, or separation between Windows and Android devices — but a single MTR VLAN is sufficient for the majority of deployments and is simpler to manage.

| VLAN | Purpose | Example Subnet |
|------|---------|----------------|
| VLAN 100 (example) | MTR room devices (Windows + Android) | 10.10.100.0/24 |
| VLAN 101 (optional) | Teams Panels and scheduling displays | 10.10.101.0/24 |
| VLAN 10 | General office workstations | 10.10.10.0/22 |

The VLAN number is site-specific — use whatever numbering convention the client organization has already established. The subnet example above is illustrative; any RFC 1918 range the organization assigns is valid.

**Subnet sizing:**

Size the MTR subnet based on room count plus growth headroom. A /24 (254 usable addresses) is appropriate for the vast majority of sites — even large campuses rarely deploy more than 200 MTR room devices at a single location. A /25 (126 usable) works for smaller sites. Avoid /26 or smaller for MTR — the administrative overhead of managing IP exhaustion is not worth the marginal subnet efficiency.

**DHCP reservations:** Assign DHCP reservations for all MTR devices based on MAC address. This ensures each device always receives the same IP address, which simplifies firewall rules (if IP-based rules are used), remote management, and troubleshooting. Document each reservation in the as-built network documentation.

---

## Inter-VLAN Routing

MTR devices require internet access to reach Microsoft 365 services. The MTR VLAN must have a default gateway configured that routes to the internet — typically through a firewall or a Layer 3 switch with an upstream path to the internet.

**Routing to internal services:** MTR devices also need to reach internal resources — NTP servers (for clock synchronization), internal DNS resolvers, and potentially internal proxy servers. Ensure that the routing table for the MTR VLAN includes paths to these internal segments, not just the internet default route.

**No requirement for lateral access to workstations:** MTR devices do not need to initiate connections to general workstation VLANs. The inter-VLAN routing policy can block lateral access between the MTR VLAN and workstation VLANs, which limits the blast radius if an MTR device is compromised. The exception is Proximity Join (Bluetooth, which operates at Layer 2 and does not require routing) and any network-based content sharing or room finder features that your organization uses.

---

## DHCP Configuration

**Required DHCP options for the MTR VLAN:**

| Option | Value | Reason |
|--------|-------|--------|
| Default Gateway | MTR VLAN gateway IP | Internet routing |
| DNS Server | Internal DNS resolver (+ public backup) | Name resolution for M365 endpoints |
| NTP Server (Option 42) | Internal NTP server | Clock synchronization for OAuth |
| Lease Time | 24 hours or longer | Stability — avoids IP changes during meeting sessions |

**NTP via DHCP Option 42:** Configure the DHCP server to deliver the organization's NTP server address via Option 42. This supplements the NTP configuration in the device's OEM admin interface and ensures NTP is available from the network layer even if device-level NTP configuration is incorrect. Android MTR devices in particular benefit from this belt-and-suspenders approach to NTP delivery.

---

## Switch Port Configuration

Each switch port connecting an MTR device should be configured as an access port (not a trunk) in the MTR VLAN. No VLAN tagging is required on the device side — the switch handles VLAN assignment by port.

**PoE considerations:** If the MTR device is powered via PoE (common for Teams Panels, Android bar devices, and some Windows compute units), the switch port must be configured for the appropriate PoE standard (802.3af, 802.3at, or 802.3bt depending on the device). Verify the PoE requirement for each device model before configuring the switch.

**Spanning tree:** Ensure PortFast (or equivalent rapid STP transition) is enabled on MTR switch ports. MTR devices do not participate in spanning tree — enabling PortFast prevents the 15–30 second STP listening/learning delay from causing network connectivity issues at device boot.

## Summary

By completing this lesson, you can now:

- **Justify segmentation:** Dedicated MTR VLAN enables QoS by source subnet, scopes firewall policy to a known IP range, and creates a manageable device segment.
- **Size the subnet:** /24 for most sites — 254 usable addresses with room for growth.
- **Configure DHCP:** Default gateway, internal DNS, NTP via Option 42, longer lease times (24h+), DHCP reservations for all devices.
- **Set switch ports:** Access port in MTR VLAN, PoE per device requirements, PortFast enabled.
