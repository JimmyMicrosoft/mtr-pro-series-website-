## Overview

Quality of Service (QoS) is the mechanism that tells the network to prioritize real-time media traffic — Teams audio, video, and content sharing — over less time-sensitive traffic like file downloads, software updates, and general browsing. Without QoS, all traffic competes equally for bandwidth. When the network is congested (which happens in real enterprise environments throughout the business day), Teams media packets wait in the same queue as backup jobs and system updates — producing the choppy audio and frozen video that users report as "Teams issues." QoS solves this at the network layer, independent of Teams application behavior.

---

## How DSCP QoS Works

QoS operates by marking each IP packet with a priority value in the DSCP (Differentiated Services Code Point) field of the IP header. Network devices (switches, routers) read this field and apply different forwarding behavior based on the value — a concept called Per-Hop Behavior (PHB).

**The DSCP field** is 6 bits in the IP header's Type of Service (ToS) byte. This gives 64 possible values (0–63), though in practice only a small number of standardized values are used.

**Per-Hop Behavior:** Each network device along the path reads the DSCP value and places the packet in the appropriate queue. Packets in the Expedited Forwarding (EF) queue are transmitted first — before packets in lower-priority queues. Packets in lower-priority queues are served when the higher-priority queue is empty or during their allotted time slice (weighted fair queuing).

The key operational point: **DSCP marking only improves quality on congested links.** On an uncongested link, all packets are transmitted immediately regardless of DSCP value. QoS is the insurance policy for when the network gets busy — not a constant performance enhancer.

---

## Microsoft Teams DSCP Values

Microsoft specifies the following DSCP values for Teams traffic, aligned with industry-standard QoS recommendations:

| Traffic Type | DSCP Value | DSCP Name | Per-Hop Behavior |
|-------------|------------|-----------|-----------------|
| Audio | **46** | Expedited Forwarding (EF) | Highest priority — always served first |
| Video | **34** | Assured Forwarding 41 (AF41) | High priority, occasional dropping allowed |
| App sharing / Content | **18** | Assured Forwarding 21 (AF21) | Medium priority |
| Signaling | **24** | Class Selector 3 (CS3) | Medium priority |

**Audio at DSCP 46 (EF)** is the most important. Voice and meeting audio is the most sensitive to latency and jitter — even 50–100ms of additional delay makes audio unintelligible. EF is the highest-priority DSCP PHB and should be reserved exclusively for real-time audio.

**Video at DSCP 34 (AF41)** tolerates slightly more variation than audio but requires consistent bandwidth. AF41 provides assured forwarding with low drop probability.

**Content sharing at DSCP 18 (AF21)** is the least time-sensitive of the three media types — a brief pause in a screenshare is noticeable but not as disruptive as choppy audio.

---

## Two-Layer QoS: Endpoint Marking and Network Enforcement

QoS requires both layers to function correctly:

**Layer 1 — Endpoint marking:** The MTR device (Windows or Android) marks outgoing packets with the correct DSCP value. On MTR Windows, this is configured through a Group Policy Object (GPO) or Intune policy that applies Windows QoS policies — marking outgoing UDP traffic on specific port ranges with the correct DSCP value. Teams uses specific source port ranges per traffic type, which allows the OS to mark correctly even without application-level awareness.

**Layer 2 — Network enforcement:** Switches and routers read the DSCP marking and place packets in the correct queue. If the network is not configured to honor DSCP markings — which is not always the default on managed switches — the markings are present but have no effect.

**What happens without each layer:**

- Endpoint marks, network doesn't honor: Packets carry DSCP values but all traffic is treated equally — QoS has no effect on quality.
- Network honors markings, endpoint doesn't mark: All Teams traffic arrives unmarked (DSCP 0 = Best Effort) and is treated as lowest priority — worst case scenario during congestion.
- Neither layer configured: All traffic competes equally — QoS provides no protection.
- Both layers configured: Teams media gets priority treatment at every hop — QoS works as designed.

---

## Port Ranges for Teams Traffic Types

Microsoft recommends specific source port ranges for each Teams traffic type. These port ranges are the basis for both endpoint QoS marking (mark traffic from these source ports with this DSCP value) and network re-marking (if the network does not trust endpoint markings, remark based on source IP and port range).

| Traffic Type | Recommended Source Port Range |
|-------------|-------------------------------|
| Audio | 50000–50019 |
| Video | 50020–50039 |
| App sharing | 50040–50059 |

These port ranges are configured in Teams Admin Center under Meeting settings → Quality of service → Insert Quality of Service (QoS) markers for real-time media traffic. Enabling this in TAC instructs Teams clients and MTR devices to use these specific source port ranges, enabling correct endpoint marking.

## Summary

- **Understand the two layers:** Endpoint marking (Windows QoS policy marks packets) + network enforcement (switches honor DSCP) — both required.
