## Overview

The network validation sign-off is the formal confirmation that the network layer is ready for MTR deployment. It consolidates all the testing and configuration verification from Modules 1.1, 1.2, and 1.3 into a single structured document that both the integrator and the client IT team sign. Producing this document before devices are installed protects both parties — it creates a clear record of the network state at deployment time, establishing the baseline for any future quality investigations.

---

## The Complete MTR Network Validation Checklist

The following checklist covers every network requirement for an MTR deployment. Each item should be tested and verified, not assumed based on design intent. Document the test result (pass/fail) and the tester for each item.

### Section 1 — Network Segmentation

| Check | Method | Pass Criteria |
|-------|--------|--------------|
| Dedicated MTR VLAN created | Switch configuration review | VLAN exists with correct VLAN ID |
| MTR VLAN assigned to correct switch ports | Switch port configuration review | All MTR ports in correct access VLAN |
| DHCP server serving the MTR VLAN | Connect device, confirm IP assignment | Device receives IP in MTR subnet |
| DHCP includes correct DNS server | Check DHCP scope options | DNS server reachable from MTR subnet |
| DHCP includes NTP (Option 42) or NTP configured at device level | DHCP scope review + device NTP test | NTP server reachable and device clock accurate |
| Default gateway functional | `ping [gateway-IP]` from device | Ping succeeds |

### Section 2 — Firewall and Connectivity

| Check | Method | Pass Criteria |
|-------|--------|--------------|
| TCP 443 to login.microsoftonline.com | `Test-NetConnection` | TcpTestSucceeded: True |
| TCP 443 to *.teams.microsoft.com | `Test-NetConnection` | TcpTestSucceeded: True |
| TCP 443 to *.office.com | `Test-NetConnection` | TcpTestSucceeded: True |
| UDP 3478–3481 to Teams media IPs | TNAT tool — UDP transport test | TNAT reports UDP packets delivered |
| Proxy bypass configured for M365 Optimize | M365 Network Connectivity Test | "Direct route" shown for Teams endpoints |
| SSL inspection bypassed for M365 endpoints | TLS certificate inspection | Microsoft/public CA shown, not internal CA |
| OEM management portal accessible | `Test-NetConnection` to OEM portal FQDN | TcpTestSucceeded: True |
| NTP UDP 123 accessible | `w32tm /stripchart` or DHCP Option 42 | NTP sync successful; device time accurate |

### Section 3 — QoS

| Check | Method | Pass Criteria |
|-------|--------|--------------|
| TAC QoS port ranges enabled | Teams Admin Center → Meetings → Meeting settings | QoS toggle enabled; port ranges configured |
| Windows QoS GPO or Intune policy deployed to MTR devices | Group Policy console or Intune device configuration | Policy shows as applied on target devices |
| Audio packets marked DSCP 46 | Packet capture filter `ip.dsfield.dscp == 46` during active meeting | DSCP 46 packets visible for audio streams |
| DSCP trust enabled on switch ports | Switch port configuration review | `mls qos trust dscp` or equivalent configured |

### Section 4 — DNS

| Check | Method | Pass Criteria |
|-------|--------|--------------|
| DNS resolves login.microsoftonline.com | `Resolve-DnsName login.microsoftonline.com` | Returns valid Microsoft IP address |
| DNS resolves teams.microsoft.com | `Resolve-DnsName teams.microsoft.com` | Returns valid Microsoft IP address |
| DNS resolves graph.microsoft.com | `Resolve-DnsName graph.microsoft.com` | Returns valid Microsoft IP address |
| DNS response time acceptable | `Measure-Command { Resolve-DnsName login.microsoftonline.com }` | < 100ms |

### Section 5 — Bandwidth

| Check | Method | Pass Criteria |
|-------|--------|--------------|
| Bandwidth capacity model completed | Model documented in project file | Required bandwidth calculated and documented |
| Measured throughput meets requirement | iPerf test from MTR VLAN | Measured throughput ≥ required bandwidth |
| Internet connection capacity confirmed | Speed test or ISP documentation | Confirmed sufficient for MTR + workstation load |

### Section 6 — Wi-Fi (if applicable)

| Check | Method | Pass Criteria |
|-------|--------|--------------|
| MTR devices on 5 GHz band | Device Wi-Fi settings | 5 GHz SSID connected |
| RSSI ≥ -67 dBm at device location | Wi-Fi analyzer | RSSI at or above threshold |
| WMM enabled on SSID | AP/Controller configuration | WMM enabled confirmed |
| No connection drops during roaming test | 5-minute connected test | Zero disconnects |

## Open Items Log

Any checklist item that fails validation should be documented in an open items log with:
- Issue description
- Recommended remediation
- Responsible party (integrator or client network team)
- Target completion date
- Resolution confirmation (signed off when resolved)

**No device deployment should proceed until all Firewall and Connectivity section items pass.** QoS and Wi-Fi items may be resolved post-deployment if they do not prevent basic device operation, but should be tracked as open items with a resolution timeline.

---

## Sign-Off Structure

The completed checklist plus TNAT results plus open items log constitutes the network readiness sign-off document. It should be signed by:
1. The integrator's network/commissioning engineer (confirming tests were performed and results are accurate)
2. The client's network/IT representative (acknowledging the network state and accepting responsibility for any open items owned by the client team)

## Summary

- **Six checklist sections:** Segmentation, Firewall/Connectivity, QoS, DNS, Bandwidth, Wi-Fi — each item tested and documented.
- **Media quality results:** TNAT packet loss, jitter, RTT against pass thresholds.
- **Open items log:** All failures documented with owner, remediation, and target date.
- **Deployment gate:** All Firewall and Connectivity items must pass before device deployment proceeds.
- **Dual sign-off:** Integrator engineer + client IT representative.
